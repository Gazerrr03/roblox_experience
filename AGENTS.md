# AGENTS.md

## Mission

Keep this Roblox experience deterministic, reviewable, and safe to integrate.
Prefer small, testable changes over wide refactors. If a task crosses gameplay,
networking, and place bootstrap at the same time, stage the work and make the
data contract explicit first.

## Vibe Framework

- This repository uses a cross-agent "vibe" framework for long-lived code
  domains: `lobby`, `run`, `maze`, and `contract`.
- Start with `references/place-parallel-development.md` for the repo-wide
  delivery model.
- Treat code domains as the primary boundary. Branches, worktrees, issues, and
  PRs coordinate the work, but they do not replace code ownership.
- When you work inside `places/lobby`, `places/run`, `places/maze`, or
  `packages/shared`, read the local `VIBE.md` first and the local `NOW.md`
  second.
- Local `AGENTS.md` and `CLAUDE.md` files are intentionally thin adapters. The
  durable source of truth lives in `VIBE.md` and `NOW.md`.

## Source Of Truth

- `packages/shared/src` is the source of truth for shared config, enums,
  schemas, remote definitions, and cross-place utilities.
- `packages/gameplay/src` is the source of truth for reusable gameplay
  logic. Keep it portable and mostly free of direct DataModel side effects.
- `packages/ui/src` is the source of truth for shared UI helpers and theme
  data.
- `places/lobby/src`, `places/run/src`, and `places/maze/src` are place-specific
  composition layers. They may depend on shared/gameplay packages, but
  shared/gameplay code must not depend on a specific place.
- `tests/src` mirrors pure logic in `packages/**`. Any non-trivial rules
  change in shared/gameplay code should come with a deterministic test update.
- `DevPackages/` is generated by Wally. Do not hand-edit it.
- `packages/` is the active runtime source tree. Treat it as authoritative
  unless a task is explicitly about a source-layout migration.
- The `contract` workstream currently lives inside `packages/shared/src/Session`,
  `packages/shared/src/Network`, `packages/shared/src/Config/SessionConfig.luau`,
  and the matching tests under `tests/src/Shared`.

## Architecture Guardrails

- Keep bootstrap scripts thin. `Bootstrap.server.luau` should mostly wire up a
  service and call `:start()`.
- Put game rules in ModuleScripts, not inline in place bootstrap scripts.
- Shared/gameplay modules should avoid top-level side effects, yielding during
  `require`, and direct writes into Workspace or Player state.
- Server code owns authoritative state, progression, loot, role assignment, and
  remote creation. Clients send intent; they do not decide outcomes.
- When adding or renaming remotes, update
  `packages/shared/src/Network/Remotes.luau`, all producers/consumers, and the
  related tests in the same change. Do not create ad-hoc remotes in random scripts.
- When changing replicated state shape, update its definition (for example in a
  relevant `Config`, `Session`, or `Util` module), all producers/consumers, and
  the related tests in the same change.
- Preserve privacy boundaries. Private role data and permission tags must stay
  filtered through `packages/shared/src/Util/Visibility.luau` or an equally
  explicit visibility gate.

## Roblox-Friendly Coding Rules

- Prefer deterministic logic for anything that affects gameplay outcome. Pass a
  seed or `Random` source explicitly instead of using hidden global randomness.
- Freeze config/enums that should be immutable. Clone before mutating derived
  copies.
- Use clear ids and centralized constants instead of repeating magic strings for
  states, channels, room kinds, or role names.
- Prefer `task.wait`, `task.defer`, or RunService-driven timing over legacy
  `wait`.
- Use `WaitForChild` when reading replicated objects that may not exist yet on
  the client.
- Keep module APIs small and explicit. Return structured tables instead of
  loosely shaped payloads when state will be shared across server/client/tests.
- Make failure modes concrete. Return typed error strings or raise explicit
  errors instead of silently ignoring invalid state transitions.

## Place And Integration Rules

- If a new folder or script tree must appear in Studio, reflect it in the
  relevant `default.project.json` file in the same change.
- Keep cross-place reuse in `packages/**`; do not copy logic between
  `places/lobby` and `places/run`.
- Treat place delivery as four lines: `lobby`, `run`, `maze`, and `contract`.
  The `contract` line owns cross-place handoffs, shared remote definitions, and
  deterministic tests.
- Keep cross-place teleports inside place-local portal adapter modules. Do not
  inline new teleport payload shaping or `TeleportService` orchestration in the
  main service flow when a portal adapter can own it.
- Use place-scoped remotes for place-local HUD/runtime state. Do not reuse a
  run-only or maze-only state channel in another place.
- Do not commit placeholder production behavior silently. If a change depends on
  real place ids, call out `SessionConfig.PlaceIds` requirements in the PR or
  task note.
- Keep binary `.rbxl` edits isolated and documented. Prefer source-driven Rojo
  changes whenever possible.

## Exceptions

- Temporary rule exceptions are only allowed for short-lived prototype work.
- Any exception must be called out in the PR description with three concrete
  items: why the rule is being relaxed, which files or systems are affected, and
  what follow-up will restore compliance.
- Exceptions must not silently weaken remote contracts, private data filtering,
  or runtime source-of-truth boundaries.

## Validation Rules

Before handing work off, run the checks that match the touched area:

1. `stylua --check .`
2. `selene .`
3. If shared/gameplay logic changed, add or update a deterministic spec under
   `tests/src/Shared`. This is the authoritative location for pure logic tests.
4. If Rojo tree or runtime wiring changed, verify the affected place boots from
   its `default.project.json` without missing-module or missing-remote errors.

## Test Strategy

- Keep tests under `tests/src/Shared` deterministic and focused on pure logic,
  state transitions, schema-aware shaping, and other behavior that can be
  validated without manual Studio interaction.
- The automated test runner is `run-in-roblox`, executed against the data model
  built from `tests/default.project.json`.
- Do not replace the real runner with placeholder CI test steps that skip
  Studio execution.
- Every shared/gameplay behavior change should still update tests and document
  any temporary manual verification needed beyond the automated coverage.

## Change Hygiene

- One PR or commit should solve one coherent problem.
- If a change belongs to a single vibe, keep the patch inside that code domain.
- If a change alters a cross-place handoff, stage `contract` first and then land
  the place-local consumers or adapters.
- Record migration or compatibility notes whenever a change alters networking,
  saved state shape, config ids, or place boot order.
- Do not mix generated dependency changes, gameplay refactors, and UI tweaks in
  the same patch unless the task truly requires it.
- When repository instructions conflict, follow direct task requirements first,
  then this file, then surrounding docs such as `README.md`.
